## What runs end-to-end today

Launch the app, land on `/crews`. The full CRUD surface for crews and runners is live and the lead invariant is enforced. Start Mission is exposed via the scratch `/debug` page only ‚Äî pick a crew, fill title/goal/cwd, and hit Start. For each crew member a PTY child is spawned with the `portable-pty` stack:

- Env set: `RUNNERS_CREW_ID`, `RUNNERS_MISSION_ID`, `RUNNERS_RUNNER_HANDLE`, `RUNNERS_EVENT_LOG`, `MISSION_CWD`, and `PATH` prepended with `$APPDATA/runners/bin` so C9's CLI will win.
- A reader thread drains the PTY, emits `session/output` Tauri events, waits the child on EOF, flips the `sessions` row to stopped/crashed, and emits `session/exit`.
- Stdin injection and kill work from the debug page. `kill` drops the master PTY, signals SIGTERM‚ÜíSIGKILL on Unix, then joins the reader ‚Äî so `mission_stop` honestly waits for the child to die before flipping the mission row to `completed`.
- `mission_start` is atomic: if any runner fails to spawn, already-spawned sessions are killed + the mission is marked `aborted` before the error returns. No phantom `running` crews.

The event log exists on disk per mission (`$APPDATA/runners/crews/{crew_id}/missions/{mission_id}/events.ndjson`) with three opening events: `mission_start`, `mission_goal`, and later `mission_stopped`. Nothing reads those yet.

## What's not wired

- **C7 event-bus watcher** ‚Äî the `notify`-backed tail + per-runner inbox projection + Tauri events for the frontend. Nothing renders the event log back into the UI yet.
- **C8 orchestrator v0** ‚Äî the rule engine that would inject `mission_goal` into the lead's stdin, route `ask_lead` signals, render `ask_human` cards, and close the `human_response` loop. The PTY spawns today, but nothing tells the lead about the goal.
- **C9 `runners` CLI** ‚Äî the binary agents would call to emit signals and messages. `$APPDATA/runners/bin` is on PATH, but the binary that path should resolve isn't built.
- **C10 mission workspace UI** ‚Äî polished feed + runner rail + xterm.js + ask-human cards, replacing the `/debug` scratch page.
- **C11 Missions list + Start Mission modal** ‚Äî the real entry point. Currently users go through `/debug`.

## Known risks going into C7+

- **Event ordering at C8 replay.** The dispatch ledger + `ask_human`/`human_response` matching need to be idempotent on mission reopen. The log-side invariants (monotonic ULIDs, no gaps, flock + repair) are solid; the orchestrator hasn't been written yet, so this is an open design point.
- **CLI binary packaging (C9).** Prepending `$APPDATA/runners/bin` works; the CLI crate that actually drops a binary there doesn't exist yet. Platform quirks to watch: macOS Gatekeeper, Linux exec bit, potential `runners-core` feature-flagging once the CLI links to it.
- **xterm.js + backpressure (C10).** The debug page dumps chunks into a `<pre>` ‚Äî fine for smoke testing but every keystroke from a real agent will pile up. The real workspace needs bounded rendering + reconnect-on-reopen.
