## What runs end-to-end today

Launching the app lands on `/crews`. The full config loop works on real data:

- **Crews:** create, list, edit, delete; lead invariant enforced both at the partial unique index (`one_lead_per_crew` on `crew_runners`) and in Rust.
- **Runners (top-level):** the sidebar `Runner` item is live. `/runners` lists every runner with running-session and open-mission counters that update via `runner/activity` Tauri events. Runner Detail (`/runners/:handle`) shows default prompt, crews-using-this-runner, activity panel, and a `Chat now` action that calls `session_start_direct` and routes to `/runners/:handle/chat` with a working PTY.
- **Missions (start/stop only, debug entrypoint):** `/debug` still drives `mission_start` / `mission_stop`. Each mission spawns one PTY child per crew member through `portable-pty` with `RUNNER_CREW_ID`, `RUNNER_MISSION_ID`, `RUNNER_HANDLE`, `RUNNER_EVENT_LOG`, `MISSION_CWD`, and `$APPDATA/runner/bin` prepended to `PATH`. `mission_start` is all-or-nothing; `mission_stop` reaps before flipping the row to `completed`.
- **Event log on disk:** every mission writes `$APPDATA/runner/crews/{crew_id}/missions/{mission_id}/events.ndjson` with monotonic ULIDs, `flock`-scoped append, on-disk floor for cross-process monotonicity, and crash-tail repair.
- **Event bus in memory:** `BusRegistry` mounts a `notify` watcher per live mission in `mission_start` and unmounts in `mission_stop` (and on rollback). The frontend can already subscribe to `event/appended` and per-runner `inbox/updated` / `watermark/advanced` — but no UI consumes them yet.

## What's left to build for v0 MVP

Four chunks remain. Order below is the recommended path; C9 and C10 can run in parallel after C8 lands.

### C8 — Orchestrator v0

**Where:** `src-tauri/src/orchestrator/mod.rs` is currently a 132-byte stub with a single TODO comment. Everything in the §C8 deliverables (`docs/impls/v0-mvp.md`) still needs to land.

**Scope reminder.**
- Built-in rules (signal-driven only, per arch §5.5.0): `mission_goal → inject @lead`, `human_said → inject @target ?? @lead`, `ask_lead → inject @lead`, `ask_human → emit human_question + open card`, `human_response → inject the original asker`.
- In-memory dispatch ledger keyed on `triggering_event_id` so replay-on-reopen is idempotent.
- Pending-ask map keyed on `question_id` so `human_response` can route back to whichever runner emitted the matching `ask_human` (lead in the lead-mediated flow, worker in the fallback).
- Subscribes to the existing `BusRegistry` — the bus already serializes events through one consumer thread, so the orchestrator gets a clean ordering guarantee.

**Open risks.**
- **Replay determinism on reopen.** Today the bus reconstructs inbox/watermark state by replaying the log; the orchestrator must reconstruct *its* in-memory state (dispatch ledger, pending-ask map) the same way. Easiest path: drive both reconstruction and live processing through the same handler, so cold-start is just "drain the file, then resume tailing". The bus already exposes the right primitives.
- **Stdin injection ordering.** `SessionManager::inject_stdin` is fire-and-forget today. If two rules fire on the same event (rare in v0 — each built-in rule is single-output) we'd need a queue. For MVP, document that injects are unordered across runners but ordered within a runner.

### C9 — `runner` CLI binary

**Where:** there is no `cli/` crate in the workspace yet. Workspace members today are `src-tauri` and `crates/runner-core` (`Cargo.toml:5`).

**Scope reminder.**
- New `cli/` workspace member producing the `runner` binary.
- Resolves envelope fields from the env vars C6 already sets (`RUNNER_CREW_ID`, `RUNNER_MISSION_ID`, `RUNNER_HANDLE`, `RUNNER_EVENT_LOG`).
- Subcommands: `runner signal <type> [--payload <json>]`, `runner msg post <text> [--to <handle>]`, `runner msg read [--since <ts>] [--from <handle>]` (emits `inbox_read` with `payload.up_to = max ULID`), `runner help`.
- Reuses `runner_core::event_log` directly — no duplicated append code.
- Bundling: drop the binary into `$APPDATA/runner/bin/` so the existing PATH prepend in `SessionManager::spawn` (`session/manager.rs:198`, `:377`) wins over any user-installed `runner`.

**Open risks.**
- **Direct-chat behaviour.** Per the C8.5 PR, direct-chat sessions deliberately do NOT set `RUNNER_CREW_ID` / `RUNNER_MISSION_ID` / `RUNNER_EVENT_LOG`. The CLI must no-op gracefully (or print an actionable error) when those vars are absent, otherwise any agent that emits an event from a direct chat will crash.
- **Packaging.** macOS Gatekeeper, Linux exec bit, and (eventually) Windows quirks. v0 explicitly skips Windows; we still need to verify the binary ends up executable when a user-installed runner CLI assembles itself in `bin/`.

### C10 — Mission workspace UI

**Where:** no `MissionWorkspace.tsx` page exists. Routes for `/missions/...` are not wired in `App.tsx` either. The `event/appended` / `inbox/updated` / `watermark/advanced` Tauri events are already broadcast and just need consumers.

**Scope reminder.** From `docs/impls/v0-mvp.md` §C10:
- `src/pages/MissionWorkspace.tsx` — feed driven by the bus events.
- `src/components/EventFeed.tsx` — message / signal / `ask_human` card variants.
- `src/components/AskHumanCard.tsx` — Approve/Deny emit a `human_response` signal; render `on_behalf_of` chain when present.
- `src/components/MissionInput.tsx` — emits `signal human_said` (not a message event), envelope `to: null`, recipient in `payload.target` (empty for broadcast).
- `src/components/RunnersRail.tsx` + `src/components/RunnerTerminal.tsx` — xterm.js bound to the existing `session/output` event stream.

**Open risks.**
- **xterm.js + backpressure.** The `/debug` page dumps chunks into a `<pre>`. Real agent output will pile up keystrokes; the workspace needs bounded rendering and a re-attach path for closed-and-reopened mission tabs.
- **Subscription scope.** Multiple workspaces could be open across missions. Current bus events carry `mission_id`; the workspace just has to filter on it. We do not yet ship multi-window — a single open workspace per mission keeps the contract simple.

### C11 — Missions list + Start Mission modal

**Where:** no `Missions.tsx` page, no `StartMissionModal.tsx`. Today the only path to `mission_start` is `/debug`. The relevant Tauri commands (`mission_list`, `mission_get`, `mission_start`, `mission_stop`) are already exposed in `src-tauri/src/lib.rs`.

**Scope reminder.**
- `src/pages/Missions.tsx` — Active / Past tabs, status dot, "pending ask" flag derived from orchestrator state (depends on C8).
- `src/components/StartMissionModal.tsx` — crew picker, title, goal textarea, cwd with `Browse…`, Advanced collapse (stubbed).
- `Start` calls `mission_start`, then routes to `/missions/:id` (which renders C10's workspace).

**Open risk.** The "pending ask" flag on the Missions list needs the orchestrator's pending-ask map to be queryable from outside the live mission. Either (a) the orchestrator persists the open-question set into the DB next to the mission row, or (b) the Missions page reconstructs it by scanning the log on demand. (a) is cheaper for the list view; (b) is fine for MVP if we keep mission counts small.

1. **C8 orchestrator** — unblocks the lead-receives-goal half of the demo path even without a polished UI.
